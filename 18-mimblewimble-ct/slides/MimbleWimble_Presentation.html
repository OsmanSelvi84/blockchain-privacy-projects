Confidential Transactions

The problem

Public blockchains expose every amount.

On Bitcoin, anyone can see who paid whom and how much. Privacy-preserving chains hide this in different ways:

Monero

Ring signatures

Hides sender, receiver, and amount via ring signatures, stealth addresses, and RingCT.

Zcash

zk-SNARKs

Hides everything with zero-knowledge proofs — at the cost of a trusted setup.

MimbleWimble

Commitments

Hides amounts with homomorphic Pedersen commitments. No addresses, no trusted setup.

In one sentence

A blockchain where amounts are hidden, yet every transaction stays mathematically verifiable.

Proposed anonymously in 2016; used in production by Grin and Beam.

The protocol

Four cryptographic moves.

01 · Hide

Pedersen commitment

A commitment locks an amount behind a random blinding factor. Hiding, and binding.

C = v·H + r·G
02 · Check

Balance the books

Commitments are homomorphic, so the network verifies balance without seeing amounts.

ΣC_in − ΣC_out − fee·H = excess·G
03 · Prove

Kernel signature

A Schnorr signature on the excess proves balance and ownership — no value created.

s·G == R + e·P
04 · Protect

Range proof

Proves every amount is non-negative, defeating the negative-amount inflation attack.

0 ≤ v < 2⁶⁴

The proposed solution

Verifying confidential transactions on Ethereum.

  • verifyKernelSignature — confirms the spender knows the kernel excess (proves ownership).
  • verifyBalance — confirms Σ inputs − Σ outputs − fee·H equals the kernel excess (no inflation).
  • secp256k1 from scratch — point add, double, scalar-mul, and modular inverse, since the EVM has no curve precompile.
  • Range proofs stay off-chain — on-chain they would cost tens of millions of gas; verified in the Python prover instead.
Inside the verifier

The no-inflation check.

Sum the input commitments, subtract the outputs and the fee term, and confirm the result equals the published kernel excess. If the amounts did not balance, an amount term would remain and the check would fail.

uint256 x = inX[0]; uint256 y = inY[0];
for (i=1; i<inX.length; i++)        // add inputs
  (x,y) = ecAdd(x,y, inX[i],inY[i]);
for (i=0; i<outX.length; i++)       // subtract outputs
  (x,y) = ecAdd(x,y, outX[i],(PP-outY[i])%PP);
if (fee != 0) {                       // subtract fee·H
  (fx,fy) = ecMul(HX,HY,(NN-fee)%NN);
  (x,y)   = ecAdd(x,y, fx,fy);
}
return x==kernelExcessX && y==kernelExcessY;

Scope & future work

Known limitations.

  • Cut-through & kernel aggregation — not implemented; this is the CT primitive layer.
  • Bulletproofs — would replace the classical range proof and shrink it roughly ten-fold.
  • Gas optimization — affine coordinates kept for clarity; Jacobian + wNAF would cut on-chain cost.
  • No P2P layer — the focus is the cryptographic core, not networking.
